Introduction

This document aims to explain the best way to work with the product and its components.

- All the best and enjoy coding.

Getting started tips

  • Do not start from scratch, use an existing asset and modify it to learn how it works.
  • Explore the sources for ideas and sample code.
  • Use Firebug or Chrome Developer Tools to find bugs on your website. Using one of those tools will help you to save time analyzing the site and finding elements structure, like classes, id or tags
    • Quick tip: open your site with Chrome, press F12 and go to console tab, reload your page and if something goes wrong you will see your page errors in red text.
  • In case of errors, someone might have seen it too, you can try a Google search for a quick fix.

Starting the app

It's recommended to run this application using a server (Apache, IIS, xampp, etc) to serve the html files and perform necessary http request.
Anyway, this project doesn't need a server but some components will not work if the application is not running from a server.

Basic server setup (requires nodejs)

This is simple solution for a basic server setup using nodejs that might help you. If you don't have installed nodejs take a look at the Build section to learn about it

  • Install the http-server (-g installs globally)

    npm install http-server -g

  • Move to the root folder of the theme (where index.html is located) and run

    http-server . -a 127.0.0.1 -p 8080

  • If everything goes fine you can now access to the app at http://127.0.0.1:8080/

Main folders explanation

dist/

This folder contains all assets necessary to run the application (images, fonts, etc) and compiled files. This files are ready to deploy on your server. This folder is also generated using Gulp.

  • css/ This folder contains the compiled stylesheets.
  • js/ This folder contains only the final build of javascript files.
  • vendor/ This folder contains vendor assets that are listed in the index file. Files are placed here using gulp-useref plugin.
  • static/ This folder contains server side files used for demonstration. In a real app your server should listen to specific address and return the content of this files (and/or others) with specific information.

src/app

This folder contains the source files organized by components. Each component can contains PUG(JADE)/HTML, SCSS and JS files.

src/node_modules

This folder contains dependencies for development installed using npm"

src/vendor

This folder contains vendor assets installed via Bower, by reading the index file Gulp will copy only those files included in the template to copy into the dist/vendor location.

Build

Important You only need to follow this instructions if you plan to use build tools.

Build Tools

Node.js is a platform built on Chrome’s JavaScript runtime for easily building fast, scalable network applications.

Gulp is a task manager, you can define different tasks to run certain commands. Those commands does the compilation job for precompiled languages (PUG/SCSS), and concatenates the JS files.

Bower is a dependency manager, it works by fetching and installing packages from all over, taking care of hunting, finding, downloading, and saving the stuff you’re looking for. Bower keeps track of these packages in a manifest file, bower.json.

The package includes under the src/ folder the file gulpfile.js, package.json and bower.json to install the required components in order to compile the source files.
.bowerrc includes directives for bower and is used to change the destination folder where files are downloaded.
.jshintrc contains directives used to ensure a well written Javascript code.

Installing tools

The following steps are intended to be an orientation guide, if you are not experienced with this you will need to learn a bit more about it from Google :)

  • To install node and npm, go to http://nodejs.org/
  • Download and install GIT for your platform from http://git-scm.com/downloads

Once you have all tools installed

  • Open a console/terminal and npm install -g gulp bower to install Bower and Gulp

  • Move to the package master/ folder, there run the following commands

    • This command will install all node dependencies. npm install.
    • This command will install vendor dependencies bower install

  • Finally run gulp serve to start the task manager.

If everything goes fine, you should see messages in your terminal telling you that all the task are done ok. Gulp will watch for file changes to compile them automatically and will open the application using your favorite browser.

Gulp configuration

If you need to change the configuration used by Gulp, mainly paths, you can do that by editing the file gulp.config.js

In such file you can also change Gulp plugin options too.

Main Gulp tasks

Gulp tasks must be executed under folder src

> gulp serve

This command will run the default task and start the built-in server using BrowserSync

> gulp build

Use this command to build the application for development. All compiled files will be placed under 'dist' folder.

> gulp serve --prod

The '--prod' flat indicated the task is executed for production. In this case gulp will compile and start a server with production ready files. Note that the built-in server is not recommended to be used for production environments.

> gulp build --prod

Use this command to build the application for production. All compiled files will be placed under 'dist' folder.

> gulp --usesourcemaps

This modifier will create source-maps when compiling assets

> gulp clean

Delete build files

Be careful the clean task will delete compiled files under 'dist' folder without any confirmation or undo option.

> gulp lint

Lint your JavaScript code

Usage

This template was developed to use Gulp as a start of the development workflow.
Gulp provides plugins to serve the app, perform automatic browser reloading after any change and build of assets for distribution. It also allows to work with agile languages like PUG/SCSS/etc and modular JavaScript (concatenated).

Beyond that, if you are not convinced about build tools, the current setup has a basic support to work with static files (html, css) without needing build tools.

We encourage you to use build tools, the benefits greatly outweigh the costs of learning.

Static files organization (no gulp)

The most simple usage is by pointing the server root the 'dist' folder of the package. Note that the the index.html included in the root of the project is just a redirect to dashboard.html.

This allows to work directly with raw files like CSS, HTML and JS without worrying about build tools.
Now, to edit files focus on the following locations:

  • Styles are accessible in folder /dist/css
  • Markup is accessible in folder /dist (raw html)
  • Scripts are accessible in folder /dist/js

Important if you change something on this folder and then use Gulp, all previous changes may be lost because Gulp uses this folder as destination to place compiled files.

Compiled files organization

As quick reference, the current setup does the following compilations:

Language Produces
HTML HTML
PUG HTML
SCSS CSS
Javascript Javascript (concatenation)

HTML files found in components folder are copied directly to the 'dist' folder

Styles

Each component contains its own styles. Gulp read all folder recursively and includes all files related to styles into one single file: app.css

The inclusion order is determined by the way file are read like follows:

path_to_components + 'app.scss',
path_to_components + '**/*.scss'

As you can see, the file app.scss is read first. Use this file to add any styles that you need before anything else. For example, @import directives.

Scripts

Each component contains its own scripts. Gulp read all folder recursively and concats its content into one single file app.js

The inclusion order is determined by the way files are read like follows:

path_to_components + 'app.js'
path_to_components + '**/*.js'

The main file app.js is used as the start point of the application. Anyway, you will see that such file only has a wrapper function, that's because each module should be responsible to start a functionality by itself.

Content of the app.js

// APP START
// -----------------------------------

(function() {
    'use strict';

    // Disable warning "Synchronous XMLHttpRequest on the main thread is deprecated.."
    $.ajaxPrefilter(function(options) {
        options.async = true;
    });

})();
Views

By default views are written in PUG (a.k.a JADE) for quick development. Using this language you will get a well organized and readable markup without need to deal with closing tags and tag delimiters.

Gulp workflow also supports raw HTML, all this files are copied directly without any special process. Only PUG files are compiled separately as HTML file.
Meaning,
file.pug compiles to file.html
file.html is copied as file.html

Index and Layout

One of the main benefits of using PUG is that we can use partials templates to write common parts of a page only once and reuse such template many times. This happens also for layout, similar to templates but allows to define 'blocks' where we will add later markup for a specific page.

Said that, the main application layout files is core.layout.pug

All application pages extends the main layout. The main layout includes all styles and scripts and defines the main structure of an HTML document.

A page that uses this layout will have at least the following content:

extends ../core/core.layout
block pagetitle
    - pageTitle = 'Page title'
block content
    .container-fluid
        // specific page content

As you can see, first we extend the layout and thus we have access to redefine blocks declared in such layout.
The 'pagetitle' block is used to display the title of the page in the header area. The 'content' block is used to set the specific content for a page.

Different from layouts, a PUG template is included in a page. In other words, a page can extend a layout and include a template. Layouts can include templates too.

As a convention we use the patterns *.layout.pug for Layouts, and *.tpl.pugfor templates.

Here is an example of the main layout content (note the included templates):

doctype html
html(lang='en')

    head
        meta(charset='utf-8')
        meta(name='viewport', content='width=device-width, initial-scale=1, maximum-scale=1')

        meta(name='description', content='Bootstrap Admin Template')
        meta(name='keywords', content='app, responsive, jquery, bootstrap, dashboard, admin')

        title Dasha - Bootstrap Admin Template

        // Vendor styles
        // build:css(./) css/vendor.css
        // Animate.CSS
        link(rel='stylesheet', href='vendor/animate.css/animate.css')
        // Bootstrap
        link(rel='stylesheet', href='vendor/bootstrap/dist/css/bootstrap.min.css')
        // Ionicons
        link(rel='stylesheet', href='vendor/ionicons/css/ionicons.css')
        // Material Colors
        link(rel='stylesheet', href='vendor/material-colors/dist/colors.css')

        block vendor_css

        // endbuild

        // Application styles
        link(rel='stylesheet', href='css/app.css')

    body.theme-default

        .layout-container
            // top navbar
            header.header-container
                include ../header/header.tpl.pug

            // sidebar
            aside.sidebar-container
                include ../sidebar/sidebar.tpl.pug
            .sidebar-layout-obfuscator

            // Main section
            main.main-container

                // Page content
                section.section-container
                    block content
                // Page footer
                footer.footer-container
                    include ../footer/footer.tpl.pug

        block bodychild

        // Search template
        include ../header/header-search.tpl.pug
        // End Search template
        // Settings template
        include ../settings/settings.tpl.pug
        // End Settings template


        // build:js
        // Modernizr
        script(src='vendor/modernizr/modernizr.custom.js')
        // PaceJS
        script(src='vendor/pace/pace.min.js')
        // jQuery
        script(src='vendor/jquery/dist/jquery.js')
        // Bootstrap
        script(src='vendor/tether/dist/js/tether.min.js')
        script(src='vendor/bootstrap/dist/js/bootstrap.js')
        // Material Colors
        script(src='vendor/material-colors/dist/colors.js')
        // Screenfull
        script(src='vendor/screenfull/dist/screenfull.js')
        // jQuery Localize
        script(src='vendor/jquery-localize-i18n/dist/jquery.localize.js')

        block vendor_js

        // endbuild

        // App script
        script(src='js/app.js')

We use gulp-useref to detect files used in the project from Bower, and copy them to the dist/vendor location. This way you can deploy only necessary files and avoid any extra assets downloaded from Bower.

Important Other files like images, fonts, etc can be configured to be included in the copy by editing the file gulp.config.js

Behavior classes

As you can see in the index, there are some classes related to the sidebar that are used to change the layout behavior.
This classes are:

Classname Description
sidebar-visible Makes the sidebar visible on mobiles and is activated only via Javascript.
sidebar-cover Enables the sidebar cover mode (offcanvas) for desktops.
footer-fixed Makes the footer always visible.
Sidebar

Sidebar related styles, markup and scripts are contained under the components/sidebar folder.

The file sidebar.tpl.pug contains the markup to generate the sidebar and each menu entry.

The sidebar uses an element (.sidebar-layout-obfuscator) to show a backdrop screen when it's open on mobiles. This backdrop also allows the user to click on almost any place to close the sidebar easily.

Header

Header related styles, markup and scripts are contained under the components/header folder.

The file header.tpl.pug contains the markup to generate the top navigation bar.

Under this module also exists the search bar scripts. The markup for the search bar can be found in file header-search.tpl.pug

In the file header.js you can find functions to show the search bar and the right sidebar. Both components are based on Bootstrap modals with minor modifications to the styles. Custom modals styles can be found in file common/modals.scss

RTL Stylesheet

Right to left feature is based on rtlcss tool, specifically the Gulp plugin version, gulp-rtlcss
RTL version of stylesheet are always generated (via GUlp) and not included in the template by default.
To enable RTL, is necessary to include the RTL variant stylesheets of the application and Bootstrap, you can do that in the following file components/core/core.layout.pug 

// Change link for Bootstrap
link(rel='stylesheet', href='vendor/bootstrap/dist/css/bootstrap-rtl.css')
// Change link for application
link(rel='stylesheet', href='css/app-rtl.css')

Note that the RTL version of Bootstrap is generated in the same vendor folder. This is done this way so the task that that build the vendor assets can use the file as if it was installed from Bower.

Important More customization might be necessary depending on plugins used. The RTL version changes the global styles and components of the template.

Translation

The translation system uses jQuery Localize This modules simplifies the translation system by loading translate references from a JSON file and replacing the content where the reference has been used.

Examples

<small data-localize="dashboard.WELCOME"></small>

The JSON files with translation references are located in the folder static/i18n

Example content of the file app/i18n/en.json

{
    "translate": {
        "EXAMPLE": "This is an example text using English as selected language."
    }
}

As a convention, we use the key name with the translated text in uppercase.

Settings and Themes

All settings related files belongs to components/settings

There you will find the initial values for the different settings supported. The template markup for the settings panel (the right sidebar) and the styles for such panel.

Themes

Theming is based on class-names that provides colors to components.

All themes classes available are configured in components/common/themes.scss

Each of this theme, named theme-* is applied to the body tag via JavaScript.

Themes also uses mixins so you can easily edit or add new themes.
Example:

.theme-4 {
    @include theme-sidebar($mdc-blue-grey-900, #fff);
    @include theme-brand($mdc-blue-grey-900, #fff);
    @include theme-header($mdc-blue-700, #fff);
    @include theme-content($body-bg);
    @include theme-sidebar-items($mdc-blue-700);
}

Note that many variables for colors are defined in file components/common/material-colors.scss

Default theme

To set a theme as default, just add the classname of the theme you want to the body tag.

Is a good practice adding the classname directly to the body so the styles are interpreted by the browser as soon as the page loads.

Customization

Here you will find information about how to add new features to the template.

Adding new Components

To add a new component create a new folder under 'src/app/components/'

For example,'components/mycomp'

Adding styles

To add a new styles just a create a new SCSS file into the component folder. This file will be automatically compiled and added into the final CSS file.

Separated components should not have conflicts between CSS rules but just in case, check the specificity in your styles (read more here)

Adding views

Create a new PUG (or HTML) file.

For example, mycomp.pug will be compiled to mycomp.html

The minimum necessary for a view markup is the .container-* element

mycomp.pug

extends ../core/core.layout.pug
block pagetitle
    - pageTitle = 'Page Title'
block content
    .container-fluid
        // here goes the content, .row, .panels., .cards, etc...

or, with a centered container 

extends ../core/core.layout.pug
block pagetitle
    - pageTitle = 'Page Title'
block content
    .container.container-lg
        // here goes the content, .row, .panels., .cards, etc...
Adding a Script

Your new component should define its own scripts in order to keep the app modular. Place in the new folder a script file with the name mycomp.js and add the following example content:

(function() {
    'use strict';

    // Run when dom is ready
    $(initMyComp);

    //////

    function initMyComp() {
        // component stuff here
    }

})();

The function that wraps the entire code is known as IIFE

Adding a Menu entry

To add a new menu, edit the file sidebar\sidebar.tpl.pug and the a new list item in the menu under element nav.sidebar-nav

Example of the Dashboard menu

li
    a(href='dashboard.html')
        span.float-right.nav-label
            span.badge-rounded.badge-primary 5
        span.nav-icon
            em.ion-ios-speedometer-outline
        span Dashboard

The first span.nav-label corresponds the float-right alert badge.
The second span.nav-icon is used for to place the menu icon.
The last span is used for the menu text (see Translation to translate this text).

Adding vendor assets

To add a new vendor assets you need to follow this steps

1- Download and install.

Usually this is done with 'bower install 3rd-party-plugin'.

All files downloaded will be placed in src/vendor/ folder

2- Add main files to the application.

Open the file where you plan to use the new plugin installed and add the styles/script:

head
    link(rel="stylesheet" href="vendor/name/some-styles.css")
body
    ......
    script(src="vendor/name/some-scripts.js")

    # application scripts

Note that we add styles in the 'head' section of the page, and scripts should placed at bottom of the 'body' but before the application scripts. This is a suggestion, you can change it as per your needs.

Adding vendor assets per page

To add vendor assets only on those pages that uses it, you can use the blocks 'vendor_css' and 'vendor_js' for styles and scripts respectively.
Take a look at the main application layout to see where are placed the mentioned blocks.
